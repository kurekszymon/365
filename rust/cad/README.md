# cad — 2D Floor Planning Engine (Rust + WebAssembly)

A 2D CAD engine built in Rust and compiled to WebAssembly. Designed for floor planning: place furniture-like objects inside a room, with collision detection and boundary enforcement enforced on every operation.

Ships as a 160 KB `.wasm` binary with a generated JavaScript/TypeScript package (`pkg/`) that can be dropped into any web bundler.

---

## Features

- **Shapes** — rectangles, L-shapes, U-shapes, arbitrary custom polygons
- **Collision detection** — objects cannot overlap; uses AABB broad phase + SAT narrow phase via `parry2d`
- **Boundary constraints** — all shapes must stay within the room boundary (which can itself be rectangular, L-shaped, or U-shaped)
- **Arbitrary rotation** — any angle, validated against both collision and boundary on every move
- **Drag preview** — `validateMove` / `validatePlacement` return a result without mutating state, enabling real-time green/red feedback
- **Point picking** — `queryPoint(x, y)` returns the shape under a cursor

## Quick Start

### Prerequisites

```bash
# Rust toolchain (1.70+)
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

# wasm-pack
cargo install wasm-pack
```

### Run the demo

```bash
# Build the WASM package
wasm-pack build --target web --out-dir pkg

# Serve locally (WASM requires HTTP, not file://)
python3 -m http.server 8765

# Open http://localhost:8765
```

### Run tests

```bash
cargo test
```

### Build for a bundler (Vite, webpack, etc.)

```bash
wasm-pack build --target bundler --out-dir pkg
```

Import in TypeScript:

```typescript
import init, { CadEngine } from './pkg/cad.js';
await init();

const cad = new CadEngine();
cad.setLShapeBoundary(12, 10, 4, 4);

const id = cad.addRectangle(2, 1, 3, 3, 0);   // width, height, x, y, rotation_deg
cad.moveShape(id, 5, 5, 45);                   // throws if invalid

const result = cad.validatePlacement(
  [[-1,-0.5],[1,-0.5],[1,0.5],[-1,0.5]],       // local polygon vertices
  5, 5, 45                                       // x, y, rotation_deg
);
// { valid: boolean, collisions: bigint[], exceeds_boundary: boolean }
```

## API Reference

All methods on `CadEngine`:

| Method | Description |
|---|---|
| `setRectangularBoundary(w, h)` | Set rectangular room, origin at (0,0) |
| `setLShapeBoundary(ow, oh, cw, ch)` | L-shaped room; `cw×ch` cutout from top-right |
| `setUShapeBoundary(ow, oh, cw, ch)` | U-shaped room; `cw×ch` notch from top center |
| `setBoundary(vertices)` | Arbitrary polygon boundary |
| `addRectangle(w, h, x, y, rot)` | Add rectangle centered at (x,y); returns `bigint` id |
| `addLShape(ow, oh, cw, ch, x, y, rot)` | Add L-shape; (x,y) is origin corner |
| `addUShape(ow, oh, cw, ch, x, y, rot)` | Add U-shape; (x,y) is origin corner |
| `addCustomPolygon(vertices, x, y, rot)` | Add arbitrary polygon |
| `moveShape(id, x, y, rot)` | Move/rotate existing shape; throws on violation |
| `removeShape(id)` | Remove shape; returns `boolean` |
| `validatePlacement(vertices, x, y, rot)` | Check placement without committing (for ghosts) |
| `validateMove(id, x, y, rot)` | Check move of existing shape, excluding self from collision |
| `queryPoint(x, y)` | Returns `bigint` id of shape at point, or `null` |
| `getSceneState()` | All shapes with world-space vertices and metadata |
| `getShapeVertices(id)` | World-space vertices of one shape |
| `getBoundary()` | Boundary vertices, or `null` |
| `exportJson()` | Serialise entire scene to JSON string |

Rotations are always in **degrees** in the public API. Converted to radians internally.

## How Collision Detection Works

### Two-phase pipeline

Every `add_shape` and `move_shape` runs the same pipeline:

```
shape A  ──► AABB overlap? ──No──► skip (no collision)
              │ Yes
              ▼
         convex sub-parts of A × convex sub-parts of B
              │
              ▼ parry2d SAT / GJK
         intersection_test(iso_a, part_a, iso_b, part_b)
```

#### Phase 1 — AABB broad phase

An **Axis-Aligned Bounding Box** is the smallest rectangle (aligned to the world axes) that contains a shape after its transform is applied. Checking whether two AABBs overlap is four comparisons:

```
overlap = (a.min.x < b.max.x) && (a.max.x > b.min.x)
       && (a.min.y < b.max.y) && (a.max.y > b.min.y)
```

If the AABBs don't overlap, the shapes definitely don't collide — the narrow phase is skipped entirely. For a scene with N shapes, this reduces O(N²) narrow-phase calls to only the small number of pairs whose AABBs actually touch.

**Cost:** O(N) per new shape (scan all existing AABBs). Suitable for up to ~100 objects. For larger scenes, upgrade to a spatial hash or BVH — both are straightforward additions.

#### Phase 2 — SAT narrow phase (via parry2d)

The **Separating Axis Theorem**: two convex polygons do **not** overlap if and only if there exists some axis along which their projections do not overlap. The algorithm tests all edge normals of both polygons as candidate axes. If any axis separates them, they're disjoint.

`parry2d` implements SAT (and the more general GJK algorithm) for convex shapes. It also handles the rotation correctly via `Isometry2` — no manual vertex transformation needed for the narrow phase.

#### Concave polygons (L-shape, U-shape)

SAT only works on **convex** polygons. L-shapes and U-shapes are concave. The solution: **convex decomposition**.

At creation time, each concave polygon is decomposed into a set of convex triangles via ear-clipping. The triangles are stored alongside the original polygon:

```
L-shape  →  triangle₁ + triangle₂ + triangle₃ + triangle₄
```

Collision between two shapes then tests all pairs of their convex sub-parts:

```
for part_a in shape_a.convex_parts:
    for part_b in shape_b.convex_parts:
        if SAT(part_a, part_b): return COLLISION
```

For simple shapes (L = 4 triangles, U = 6 triangles), the number of pairs is small and the overhead is negligible.

#### Boundary containment

The room boundary check is separate from shape-vs-shape collision. A shape is **contained** in the boundary if:

1. Every vertex of the (transformed) shape lies inside the boundary polygon — checked via ray casting.
2. No edge of the shape crosses any edge of the boundary — checked via segment intersection.

Both conditions are required because a concave boundary (L-room, U-room) can have "forbidden" regions where the first test alone would pass for a shape straddling the concave notch.

## Extending to 3D and AR

The current 2D setup uses `parry2d`, `nalgebra::Point2`, and `nalgebra::Isometry2`. Moving to 3D requires replacing these throughout. Nothing in the current design prevents it — the 2D code is not intertwined with the WASM layer — but it is a real rewrite of the geometry and collision modules.

### Effort breakdown

| Layer | 2D today | 3D extension | Change |
|---|---|---|---|
| Geometry primitives | `Point2`, `Polygon` (flat vertices) | `Point3`, `Mesh` (triangulated surfaces) | Full rewrite |
| Collision detection | `parry2d` (SAT, GJK, convex decomp) | `parry3d` (same API, 3D shapes) | Swap crate, rewrite shapes |
| Boundary containment | Ray casting + edge intersection | Ray casting against mesh faces | Rewrite |
| Transform | `Isometry2` (x, y, θ) | `Isometry3` (x, y, z, quaternion) | Swap type |
| Scene management | `HashMap<u64, PlacedShape>` | Same | Reuse |
| Validation logic | `validate_for` | Same pattern | Reuse |
| WASM API surface | Current `CadEngine` | New/extended `CadEngine3D` | Mostly reuse |
| **Total new code** | baseline | **~60% of codebase** | |

### AR (WebXR) on top of 3D

AR requires rendering, which this library deliberately does not do — it only returns world-space vertices. The frontend is responsible for rendering. Adding AR means:

| Task | Notes |
|---|---|
| WebXR session setup | `navigator.xr.requestSession('immersive-ar')` |
| Camera pose tracking | Provided by WebXR; feed into renderer |
| 3D renderer | Three.js, Babylon.js, or raw WebGL |
| Plane detection | WebXR `hit-test` module — maps real floor to world origin |
| Anchors | Persist shape positions relative to detected planes |
| **Additional effort** | **~30% on top of the 3D layer** |

The `Scene` API does not need to change for AR — the frontend simply reads `getSceneState()` and renders shapes in world space using whatever 3D renderer it chooses.

### Recommended path

1. **Now**: ship 2D floor planning as-is.
2. **3D**: create `cad-3d` as a separate Rust crate alongside this one, sharing only the `Scene` / validation pattern. Do not try to make one crate generic over 2D/3D — the shape representations are too different.
3. **AR**: add a WebXR integration layer in JavaScript on top of the 3D WASM package. The Rust side stays unchanged.
